Security-First Development: What It Actually Looks Like in Healthcare

The real decisions behind our security posture on VOSS — not the buzzwords.

← Back to Blog

Every health tech company says they take security seriously. It's on every landing page, in every pitch deck, somewhere between "HIPAA compliant" and "enterprise-grade encryption." But what does it actually mean when you're building an application that handles real patient data?

When we started building VOSS — a neurological examination tool that records patient scores, visit histories, and clinical progression over time — we had to answer that question for real. Not in a marketing document. In code.

The First Question: What Do We Actually Need?

Before writing a single line of security code, we asked a more fundamental question: what data do we actually need to collect?

The answer, it turned out, was less than we initially assumed. VOSS helps clinicians score neurological examinations and track changes over time. That requires storing examination scores, timestamps, and enough patient identifiers to link visits together. It does not require storing full medical histories, billing information, or anything else that doesn't directly serve the clinical workflow.

First Principle

Don't collect what you don't need. The most secure data is data that doesn't exist on your servers.

On-Device by Default

VOSS stores patient examination data on the device by default. Not on our servers. Not in a cloud database. On the physician's iPhone, encrypted with the device's native security capabilities.

This wasn't the easy choice. Cloud-first would have been simpler architecturally. It would have made syncing across devices trivial. It would have given us analytics capabilities that could inform product decisions.

But here's the thing: clinicians didn't ask for cloud sync. They asked "where is my data stored?" and "who else can see it?" When we said "on your phone, encrypted, and nobody else," the conversation moved to features. Trust was established. That trust was worth more than any architectural convenience.

Treating Every API Call as Hostile

For the features that do require network communication — software updates, optional data export, anonymized aggregate analytics — we follow a zero-trust model:

  • Certificate pinning. The app only communicates with our verified servers. Man-in-the-middle attacks are blocked at the network layer.
  • Token rotation. Authentication tokens have short lifespans and are rotated automatically. A compromised token has a brief window of utility.
  • Request signing. Every API call is signed to prevent tampering. If a request is modified in transit, it's rejected.
  • Minimal payloads. API responses contain only what the client needs. No full database dumps, no extra fields "just in case."

This is more work than a standard REST implementation. But in healthcare, "standard" isn't good enough. Every network interaction is a potential attack surface, and we treat it accordingly.

What Happens When a Phone Is Lost?

This was the question neurologists asked most. Not "what encryption algorithm do you use?" but the practical, concrete scenario: "I left my phone in a cab. Now what?"

Our answer has multiple layers:

  • App-level authentication. VOSS requires biometric or passcode authentication independent of the device lock screen. Even if someone gets past the iPhone's lock, they can't access the app.
  • Encrypted storage. All examination data is encrypted at rest using the device's Secure Enclave. Access to the physical storage doesn't yield readable data.
  • No cloud breadcrumbs. Since data lives on-device by default, there's no cloud account to compromise. A lost phone doesn't create a cascading breach.
  • Remote wipe compatibility. VOSS's data is included in the standard iOS remote wipe capability. If the physician uses Find My to erase their device, examination data goes with it.

"Good. Now show me how the scoring works." — the best possible outcome. Security that's thorough enough to satisfy a clinician but invisible enough to stay out of their way.

The Privacy Architecture We Chose

We describe our approach as zero-knowledge where possible, minimal-knowledge everywhere else.

Zero-knowledge means that for the core clinical data — examination scores, patient identifiers, visit history — we literally cannot read it. It's encrypted on-device with keys we don't have access to. Even if our servers were breached, there's nothing useful to exfiltrate because the data was never there.

For the optional features that do involve our servers (aggregate analytics, for example), we apply minimal-knowledge principles. Data is anonymized before it leaves the device. We can see that "a user completed 12 examinations this month" but not whose examinations they were or what the scores revealed.

What Security-First Actually Costs

Let's be honest about the trade-offs. Building this way is slower. It constrains feature design. There are product ideas we've shelved because they would require data access that contradicts our security posture.

Trade-off

A cloud-first approach would let us build collaborative features faster, offer seamless multi-device sync, and collect richer product analytics. We know this. We chose differently anyway. Because in healthcare, security isn't a feature — it's the foundation. Get it wrong and nothing else matters.

We'd rather ship fewer features with absolute confidence in their security than ship everything with fingers crossed.

Looking Forward

As VOSS grows, our security architecture will evolve with it. We're exploring privacy-preserving computation techniques that could enable collaborative research without ever exposing individual patient data. We're evaluating additional compliance frameworks as we expand into new jurisdictions.

But the core principles won't change: collect less, store locally, encrypt everything, and earn trust through transparency — not marketing claims.
